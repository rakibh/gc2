IT MANAGEMENT SYSTEM - PRODUCT REQUIREMENTS DOCUMENT (PRD)
===========================================================
Version: 2.0 (Final)
Status: Approved for Development
Date: 2026
Language: English (UI supports English & Bengali)

1. SYSTEM OVERVIEW
------------------
A comprehensive web-based IT management system for tracking equipment, 
network infrastructure, task assignment, and system maintenance in an 
office environment.

Core Features:
• Multi-user role-based access (Admin / Standard User)
• Dynamic Equipment Inventory with custom JSON schema form builder
• Network Infrastructure Management with equipment linking
• Task Assignment & Tracking with extended status workflow
• Real-time Notification System (AJAX polling)
• Administrative Tools (Backup, Optimize, Logs, Settings)

2. TECHNOLOGY STACK
-------------------
Frontend:
• HTML5
• Tailwind CSS (Play CDN) for runtime utility generation
• Alpine.js (local) for reactive UI state management
• Font Awesome Free 6 (CDN) for icons
• Inter font family
• Vanilla JavaScript + Fetch API for AJAX operations
• Note: CDN allowed for Tailwind & FA as per preference. Local fallback optional.

Backend:
• PHP 8.3.14 (strict_types enabled, PDO, Repository pattern)
• MySQL 9.1.0 (InnoDB, utf8mb4)
• Apache 2.4.62.1
• Configuration via .env (gitignored)

3. GLOBAL REQUIREMENTS
----------------------
3.1 Design & UX
• Layout: Header (dark, logo, profile, notifications) + Sidebar + Main
• Responsive: Mobile-first, Tailwind breakpoints
• Language: English & Bengali support for all UI text
• Timezone: GMT+6 (Asia/Dhaka)
• Date Format: dd/mm/yyyy
• Time Format: 12-hour with AM/PM (hh:mm:ss)

3.2 Security Standards
• PDO prepared statements only (no raw queries)
• CSRF token validation on all POST/AJAX requests
• Input sanitization & validation (client + server)
• Session: httponly=1, samesite=strict, regenerate_id() on login
• Role-based access control on every module entry
• File Uploads: MIME + extension double-check, rename on save, size limits
• Password Storage: MD5 hash (Explicitly noted as dev/test risk; upgrade to password_hash() recommended for production)

3.3 Session & Auth Rules
• Session Lifetime: 24–48 hours
• "Remember Me": Optional checkbox
• Browser Close: Session cleared | Tab Close: Session persists
• Login Errors: Generic messages only ("Invalid credentials")
• Rate Limiting: Max 5 attempts/IP+ID per 10 min → exponential lockout (10m → 30m)

4. MODULE 1: AUTHENTICATION & USER MANAGEMENT
----------------------------------------------
4.1 Login System
• Fields: "Employee ID / Username", Password (show/hide toggle)
• Validation: ID required, Password min 6 chars (1 letter, 1 number, 1 special)
• Security: CSRF, rate limiting, session regeneration, generic errors
• Redirect: Admin → dashboard_admin.php | User → dashboard_user.php
• Logout: session_destroy(), cache clear, redirect to login

4.2 User Management

a) Roles & Permissions
• Admin: Full system access, manages all users, views audit logs, admin-only tools
• Standard User: Can only view/edit own profile (Role, Employee ID, Status locked)

b) Main Interface (list_users.php)
• Access: Admin Only
• Layout: Responsive table with pagination (default: 20 records/page)
• Columns: Profile Photo, Name, Employee ID, Email, Phone, Role, Status, Actions
• Features: Shorting (Role, Status), User List Export capability
• Interaction Model: All CRUD operations (Add, View, Edit, Delete, Reset Password) are executed via Modal Dialogs triggered from this page (no separate page navigation)

4.3 User CRUD & Modal Workflows 

a) Add User Modal
• Fields: Username, Employee ID, Password, Role, Status 
• Validation: Unique constraints on ID/Username. Password policy enforced.
• Action: Hash password, save to DB, log audit entry, refresh list

b) Edit User Modal
• Access Control: Admin edits all fields. Standard User edits only allowed personal fields.
• Pre-fill: Loads existing data
• Photo Handling: Upload new, remove, or keep existing
• Photo Upload: Max 5MB, formats JPEG/JPG/PNG. user can resize photo when upload ratio will be 1:1.
• Password Change: Optional, must meet policy if updated
• Validation: Same as Add, with conflict checks

c) View User Modal
• Layout: Read-only display of all user fields
• Revision History: Collapsible section showing last 15 revisions (referance design : https://tailwindcss.com/plus/ui-blocks/application-ui/lists/feeds#component-a70becc0d3cc548c2205920a6aa78e94)
  - Columns: Timestamp, User Responsible, What Changed (e.g., "Role changed to Admin")

d) Delete User Modal
• Security: User must type "DELETE" to confirm
• Safeguard: Cannot delete own account
• Action: Permanent record deletion, cascade cleanup of related logs/assignments
• Audit: Logged before execution

e) Reset Password Modal
• Workflow: Admin enters new password → Policy check → Hash & Save
• Post-Reset: System forces user to change password on next login
• Notification: Optional alert to user upon reset

f) Profile Management (profile.php)
• Access: All authenticated users
• Read-Only Fields: Employee ID, Role, Status
• Editable Fields: Profile Photo, First/Last Name, Designation Username, Email, Phone, Password
• Photo Upload: Same limits as admin modals (5MB, JPEG/JPG/PNG)
• Password Change: Requires current password + new password meeting policy
• Feedback: Success/error alerts via Alpine.js toast/modal system

5. MODULE 2: TASK MANAGEMENT
-----------------------------
5.1 Tab Layout & Counters and Board Structure & Status
Part 1 : Bord (Design Referance- C:\wamp64\www\gc2\TEMP\part1_task.jpg)
• To Do (Count): Active tasks not yet started
• Doing (Count): Tasks currently in progress
• Past Due (Count): Tasks that have exceeded their deadline

Part 2 : Tab (Design Referance- C:\wamp64\www\gc2\TEMP\part2_task.jpg)
• All (Count): Where displaying all five types task
• Done (Count): Successfully completed tasks
• Dropped (Count): Cancelled or abandoned tasks

5.2 Status Flow Rules
• Mandatory Progression: A task MUST transition to "Doing" before it can be marked "Done".
• Auto-Flagging: The system automatically moves tasks from "To Do" or "Doing" to "Past Due" once the deadline timestamp passes.
• Drop Permissions: Any involved party (Task Creator, Assignee, or Admin) can mark a task as "Dropped" from any active status.
• Tab Visibility: "Done" and "Dropped" tasks are strictly excluded from the "To Do" tab.
• Dependencies: Explicitly NOT IMPLEMENTED (per requirements).

5.3. Core Features and Capabilities
a) Task Creation & Assignment Modal 
• Create tasks with Task Title, description, deadlines, priority, and recurrence patterns (Recurring: N/A / Daily / Weekly / Monthly patterns).
• If recurrence patterns select then need choose to Time->Daily, Day->Weekly, Date->Monthly
• Assign to one or multiple employees or ALL simultaneously. (Design Referance- https://tailwindcss.com/plus/ui-blocks/application-ui/forms/select-menus#component-71d9116be789a254c260369f03472985)
• Admin/Task Creator-only capability to extend deadlines post-creation.
• Assignee start task he/she can click a button "Start/Doing"

b) Priority Levels
• Four-tier priority system: Priority: Low / Medium / High / Urgent (color-coded)
• Color-coded badges applied across all views and lists.

c) Recurring Tasks
• Supported patterns: Daily, Weekly, Monthly.
• System auto-generates next occurrence upon completion or manual trigger.
• Recurrence metadata stored for tracking and reporting.

d) Attachments
• Supported formats: PDF, JPG, PNG, JPEG, DOC, DOCX, TXT, XLS, XLSX, CSV, PPT, PPTX, PPS, PPSX, HTML, HTM, XML
• Max file size: 5MB per attachment
• MIME + extension double-validation enforced on upload.
• Stored in secure directory with renamed filenames to prevent collisions.

e) Activity Log & Audit Trail
• Immutable log tracking: Who changed what, when, and previous vs. new values.
• Visible in task detail view for transparency and accountability.

f) Bulk Actions
• Multi-select support for updating status, reassigning users, or dropping tasks.
• Executes via AJAX with transactional safety.

5.4 Task Pages & UI Components
a) List View (list_tasks.php)
• Responsive layout (mobile-first Tailwind CSS breakpoints).
• Stateless sorting/filtering via URL parameters (zero DB overhead for UI state).
• Dynamic tab switching with live counters.
• Color-coded priority badges, status indicators, and assignee avatars.

b) CRUD Operations (Add/Edit/View/Delete)
• Rendered via Alpine.js reactive modals or slide-over panels.
• Add/Edit: Pre-filled forms, real-time validation, attachment upload zone, recurrence selector.
• View: Read-only display with activity log and attached files grid and comment section.
• Delete: Permanent removal with cascade cleanup of attachments, logs, and notifications. Requires confirmation modal (must type "DELETE" to confirm)
• Validation: CSRF tokens, input sanitization, required field checks, date/time logic enforcement.

5.5 Search, Filter, Sort & Export
a) Filtering
• Date Range (Created/Updated/Deadline)
• Priority (Low/Medium/High/Urgent)
• Status (All active tabs or custom status dropdown)
• Assignee (Single or multi-select dropdown)

b) Sorting
• Default: Due Date (ASC)
• Alternatives: Creation Date, Priority Level, Status
• Stateless: Handled via GET parameters; no UI state saved to DB.

c) Search
• Free-text search across: Title, Tags, Description, Assignee Names
• Real-time AJAX debouncing to prevent excessive queries.
• Match highlighting in results.

d) Export
• Supported formats: CSV, XLS, XLSX, TXT only
• Export respects currently active filters, sorting, and search parameters.
• Exports only visible/selected columns for data consistency.

5.6 Dashboard & Reporting 
a) Admin Dashboard
• Total tasks created (system-wide)
• Pending vs. Completed ratios
• Performance metrics: Completion rate per user/team
• Overdue tasks summary with quick-action links
• Exportable aggregated reports

b) Standard User Dashboard
• Personal task summary (Assigned vs. Created)
• "My Tasks" quick view across all statuses
• Daily/Weekly productivity stats
• Completion percentage trend
• Upcoming deadlines (next 7 days)

5.7 Notification Integration
Trigger Events
• Task Assigned: All assignees notified
• Status Changed: Creator + Assignees notified
• Deadline Approaching/Passed: Assignees + Creator notified ("Past Due" alert)
• Task Completed: Creator + Assignees notified
• Task Dropped/Cancelled: All involved parties notified

6.5 Validation Rules

a) Equipment Type Validation
- Name: Required, 2-99 characters, alphanumeric + spaces
- Slug: Auto-generated from name, lowercase, hyphens allowed
- Form Schema: Valid JSON structure

b) Equipment Validation
- Label: Required, 2-99 characters
- Type: Required, must exist in equipment_types table
- Serial Number: Optional, must be unique if provided
- Brand/Model: Optional, 0-50 characters each
- Status: Required, must be valid enum value
- Location: Optional, 0-50 characters
- Department: Optional, 0-50 characters
- Assigned To: Optional, 0-50 characters
- Warranty Dates: Optional, valid date format (YYYY-MM-DD)
- Expiry Date: Must be >= Purchase Date (if both provided)

c) Network Validation
- IP Address: Required if network enabled, valid IPv4 format
- MAC Address: Optional, valid MAC format (XX:XX:XX:XX:XX:XX)
- IP Address: Must be unique in network_info table
- Port Numbers: Optional, 0-20 characters

d) File Upload Validation
- Allowed Types: PDF, JPG, JPEG, PNG
- Max Size: 10MB per file
- Max Files: 10 files per equipment max
- Storage: /public/uploads/warranty/
- Naming: Unique filename (UUID + original extension)

10. DATABASE SCHEMA OVERVIEW
-----------------------------
Core Tables & Purpose:
• users: Auth data, roles, status, password (MD5)
• audit_logs: Action tracking (user, target, old/new JSON values)
• equipment_types: Dynamic form schemas (JSON)
• equipments: All equipment records, custom_data (JSON), warranty dates
• network_info: IP/MAC, patch/switch details, remarks
• equipment_network_map: Pivot table linking equipment ↔ network
• tasks: Task records, status, priority, recurrence
• task_assignees: Multi-user assignment tracking
• task_attachments: File uploads for tasks
• notifications: Global event logs
• user_notifications: Per-user read/acknowledge status
• system_settings: Config key-value store
• system_logs: System activity & error logs

Key Design Decisions:
• JSON for dynamic equipment data & form schemas
• Pivot table for flexible network linking
• Stateless UI: Sorting/filtering via URL params only
• User Preferences: ALL users (Admin + Standard) use localStorage only (zero DB overhead)
• Indexing: Standard indexes on dates/status/type_id; Generated columns for frequent JSON filters

11. IMPLEMENTATION & SECURITY GUIDELINES
-----------------------------------------
